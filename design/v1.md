## 卖点

1. 本质：属于自己的个人微信
2. 特色：
   3. 数据全自有
   4. 启动快速

看起来相比微信，或者说通用 IM，真的没有啥特别的优势
还是要向 log 方向多增加一些特别的 feature；或者增加一些“格调”！

## 纠结的点

- 有一天自己在纠结，消息是否需要让用户考虑“整合“—消息是离散的，但是记录需要被整理。—需要有一个整理的能力
  - 好好给出自己的“整理“建议：用 AI，给出 summary 以及“建议“：核心出发点就是，AI 可以基于信息，发现自己没有发现的东西，给出自己的建议。她之前在记录黄车车情绪的时候，发现了 AI 的帮助。这是她觉得做这个 app 最特别的点

## Features

基于 [concept][./concept.md], 规划 v1 版本 features 如下：

1. 核心功能上：
   1. <del>只做 local, 不做 sync；</del> 还是要做个人的跨端同步吧——因为这极大影响数据库和相关的操作？
   2. 加密功能不做
   3. 不做 tag 功能
   4. 不做标记功能
2. 页面上：
   1. 完成 「记录」 Tab 的核心功能
      1. 主题列表
         1. 长按出现菜单：
            1. 置顶（置顶此主题）、设置（跳转到下面的设置项页面）
      2. 消息页 (timeline)：
         1. 设置项，完成如下功能
            1. 个性化：设置主题内显示的用户名，主题背景、主题 icon、主题字体
            2. 主题编辑：修改主题名字，置顶主题、删除主题、归档主题
         2. 消息列表：支持展示文字、图片、视频、音频。
            1. 文字交互：复制
            2. 图片交互：点击全屏
            3. 视频交互：显示播放按钮、时长；点击播放时气泡内播放，显示时间轴，支持拖动、倍速；暂时不持久化播放进度
            4. 音频交互：显示播放按钮、时长；点击播放时气泡内播放，显示播放进度，支持拖动、倍速
         3. 发送框：
            1. 默认框：点按是文字输入（调起输入法），长按是语音输入（但也有单独的按钮来切换到语音输入形态）
               1. 按住做语音输入时，有屏幕交互选项：往下是取消，往上是 Lock（持续录音）；往上激活后，屏幕变成一个红色录音按钮，点击停止，然后发送
            2. 有「+图片/视频」的按钮；有「+文件」按钮；
   2. 查看 Tab
      1. 最小实现：展示当前记录的总天数、条数
   3. 应用设置页面（「设置」抽屉）：
      1. 个人资料（用户头像、用户名、ID）
         1. 头像、用户名点击后可以直接修改
      2. 关于 ImLog： 点击后展示 app 介绍、版本信息
      3. 反馈：点击后展示一个页面，里面有一个表单，用于用户提交反馈信息（表单项：用户邮箱、反馈描述、【可选】截图）
3. 数据
   1. 应用数据库：SQLDelight + SQLite
   2. 数据存储目录：
      1. 应用数据库：应用私有位置，app-specific storage https://developer.android.com/training/data-storage/app-specific；随便读写，卸载后自动清除
      2. 用户的原生消息数据：Shared Storage, Documents/Download 数据地址，见 https://developer.android.com/training/data-storage/shared/documents-files 需要用户手工选择位置，在此基础上，应用不需要申请额外的权限！卸载后会被保留
   3. 备注：因为应用数据库卸载后被清除；所以如果用户重装 app，可以考虑从用户原生消息数据里重建应用数据库；应该还是比较容易的（但需要保存额外的 meta 数据）

## 数据库定义

### 建表的基础考量

- 要考虑到这个表信息是需要被同步分发到 topic 关联的用户，还是只分发到当前 user 的跨端设备，还是仅设备本地。
- 单设备多用户登录：
  - app 层面的全局信息，放到  Jetpack DataStore (Preferences) 中
  - 用户层面的信息，一人一库：
    - 通过对各个 user-id 建立独立的顶级文件夹来区分不同用户；所以各用户在设备维度是完全独立的
    - 也就是说，**下面所有的 Table, 都是在 user-id scope 下的**
- App meta 数据加载的流程
  
  将系统元数据分为 “启动引导数据 (Bootstrap Data)” 和 “数据库元数据 (Database Metadata)”。
  App 启动
    │
    ├── 1. 读取 DataStore/SharedPreferences (轻量、极速)
    │      └── 获取当前登录的 user_id, 确定 App 版本
    │   
    ├── 2. 初始化 SQLite 数据库 (根据 user_id 打开对应的 DB 文件)
    │
    └── 3. 读取 SQLite 中的 device_metadata 表 (获取数据库相关的同步状态、设备ID等)
   
  在 Android 原生开发（Jetpack Compose）中，这类轻量级的全局配置（K-V 键值对）推荐使用 Jetpack DataStore (Preferences)。
  
  为什么在这里存？
  启动速度极快：无需初始化繁重的 SQLite 驱动和 ORM 框架，App 启动瞬间即可读取。
  解决多账号初始化问题：存储当前活跃的 current_user_id，App 读取后，去加载 /databases/user_{user_id}.db。
  
  存储内容示例：
  active_user_id (TEXT): 当前登录的用户 ID。如果为空，则跳转到登录/扫码界面。
  last_app_version_code (INTEGER): 上次运行时的 App 版本号（用于判断 App 是否刚刚经历了应用商店的升级，从而执行某些一次性的迁移逻辑）。
  global_theme (TEXT): 全局主题偏好（跟随系统、深色、浅色）。


### app 信息（放到 Jetpack DataStore)


1. App 全局 DataStore: global_configs

  current_user_id

  theme_mode（深色/浅色模式）： 取值：SYSTEM（跟随系统）、LIGHT、DARK。
  - 必须在 Activity 渲染第一帧前读取，否则会导致界面从白闪烁变黑。
  - 作为牺牲，就将其放到全局，而不是逻辑上更合理的 user-specific 
  - 当然，可以直接选择为系统（同理，显示语言也可以基于系统）
  - 因为一般来说，单独设置 app 维度的 theme/lang 的概率太低，为了这个担心实在不值当
  last_run_version_code

1. user specific DataStore
   is_onboarding_completed
   default_input_method: keyboard / voice (没必要同步，不同设备偏好可能不同)


### device_metadata 表

CREATE TABLE device_metadata (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL
);

在 Local-First 架构中，它主要用来记录与这个数据库文件生命周期强绑定的状态：

Key (键)	Value (值)	作用说明
local_device_id	TEXT (UUID)	当前设备在这个数据库中的唯一标识（用于向同步服务器表明“我是设备 A，请发给我 A 还没收到的变更”）。
last_sync_timestamp	INTEGER	上次成功与服务器同步的时间戳。
database_created_at	INTEGER	本地数据库首次初始化的时间。
is_database_initialized	INTEGER (0/1)	本地基础数据（如默认主题、内置标签等）是否填充完成的标记。

### 用户表 (users)

记录当前本地用户以及通过同步或扫码加入的其他协同记录者/观察者的基本信息。

字段名	类型	约束	说明
id	TEXT	PRIMARY KEY	用户唯一标识 (UUID)
username	TEXT	NOT NULL	用户名
avatar_uri	TEXT		头像的本地路径或远程 URL
created_at	INTEGER	NOT NULL	创建时间（Unix 毫秒时间戳）
updated_at	INTEGER	NOT NULL	更新时间（Unix 毫秒时间戳）；for future sync

### 主题表 (topics) 及关联表


`topics` - 对应“记录 (Tracing)”Tab 中的主题列表。

存储 meta 信息；未来需要同步给所有的 members

字段名	类型	约束	说明
id	TEXT	PRIMARY KEY	主题唯一标识 (UUID)
name	TEXT	NOT NULL	主题名称
icon_uri	TEXT		主题图标的本地/远程路径
creator_id	TEXT	REFERENCES users(id)	创建者 ID
created_at	INTEGER	NOT NULL	创建时间
updated_at	INTEGER	NOT NULL	更新时间; ；for future sync
is_deleted	INTEGER	DEFAULT 0	软删除标记 (0=正常, 1=已删除)

索引：creator_id, is_deleted

`topic_members` – 主题成员表 
   
记录用户与主题的关系及角色权限; 这张表需要在此 topic 下同步给所有的 members

字段名	类型	说明
topic_id	TEXT NOT NULL	主题ID
user_id	TEXT NOT NULL	用户ID
user_nickname	TEXT		成员在该主题内的个性化显示名称
role	TEXT NOT NULL	角色：owner / contributor / observer
joined_at	INTEGER NOT NULL	加入时间
updated_at	INTEGER NOT NULL;  for future sync
is_deleted	INTEGER	DEFAULT 0	软删除标记（用于移除成员的同步）

联合主键	PRIMARY KEY (topic_id, user_id)		
外键：topic_id → topics(id) ON DELETE CASCADE，user_id → users(id)
索引：topic_id, user_id


`topic_personal_state` – 个人层面的 topic 信息/状态

存储单个用户跨设备同步的状态。仅在单个用户维度同步。

字段名	类型	说明
topic_id	TEXT NOT NULL	
user_id	TEXT NOT NULL	
archived	INTEGER DEFAULT 0	是否归档（不再显示在记录 Tab）
pinned	INTEGER DEFAULT 0	是否置顶
background	TEXT	背景图片路径或颜色值； 将 background 字段保留在 topic_user_sync_settings 中。文件本身不通过 PowerSync 传输，由用户自行配置的文件同步工具负责。App 在加载背景时，若路径对应的本地文件不存在，可显示默认背景或提示用户重新选择。
icon	TEXT	图标路径或资源标识
last_read_at	INTEGER	该用户在此主题中最后阅读的时间（用于计算未读）
updated_at	INTEGER NOT NULL	

PRIMARY KEY：(topic_id, user_id)
外键：(topic_id, user_id) 参照 topic_members（应用层保证）
索引：(topic_id, user_id), archived, pinned

`topic_device_state` – 本地设备维度的状态（不同步）

存储纯设备本地的偏好，当前只有字体。不同步，不纳入 PowerSync。

字段名	类型	说明
topic_id	TEXT NOT NULL	；PRIMARY KEY
font	TEXT	用户在此主题中偏好的字体名称（仅本设备）

外键：topic_id

注意，这里没有存储 user_id, 因为在 device 维度只有当前 user_id. 


### messages – 消息/记录表
存储每条记录的核心内容。支持文本、图片、视频、音频、文件类型。

字段名	类型	说明
id	TEXT PRIMARY KEY	
topic_id	TEXT NOT NULL	所属主题
sender_id	TEXT NOT NULL	发送者用户ID
type	TEXT NOT NULL	text / image / video / audio / file
content	TEXT	文本消息的内容；对其他类型可存描述或空
file_path	TEXT	资源文件的本地相对路径（非文本消息）
file_size	INTEGER	文件大小（字节）
duration	INTEGER	音视频时长（秒）
thumbnail_path	TEXT	视频缩略图路径（可选）
created_at	INTEGER NOT NULL	发送时间
updated_at	INTEGER	最后编辑时间（与 created_at 相同表示未编辑）
is_deleted	INTEGER DEFAULT 0	软删除标记（仅自己的消息可删除/撤回）

外键：topic_id → topics(id) ON DELETE CASCADE，sender_id → users(id)
索引：
- (topic_id, created_at) – 用于拉取主题消息列表 （这个联合索引加速排序过程）
- (sender_id, created_at) – 用于筛选“记录人”
- (created_at) – 用于日历视图按日期排序
- type – 用于按文件类型筛选
