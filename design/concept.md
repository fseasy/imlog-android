## 核心目标

基于 IM 的形式，实现一个多主题的个人记录 First, 支持多人共同记录的软件。

App 核心特点：

1. 启动快：启动需要轻量、快速，不错过瞬间的思绪
2. 可记录文字、声音（语音）、图片、视频
3. 记录可以放到不同的主题（类似聊天群）；也可以在一个主题内通过给消息打 Tag 来进一步归类记录
4. 记录数据完全属于用户自己：数据文件本地存储（可加密/非加密，加密的话使用非特有的加密方案，用户可自己解密，不依赖 app），用户可自己通过第三方软件同步这些数据（如 apple cloud 等）
5. 本地优先，可选同步、共同记录

### 加密的细节

需要解决的三个安全问题

| 场景                               | 要保护什么                                     | 谁来解密                                         |
| ---------------------------------- | ---------------------------------------------- | ------------------------------------------------ |
| 文件存放在本地（手机里）           | 照片/语音/文本文件不能被别的应用直接读取       | 这台手机上的 app（确切地说，是拥有密钥的「你」） |
| 文件同步到云端（iCloud/WebDAV 等） | 云服务商看不到内容                             | 所有主题成员的设备                               |
| 新成员加入                         | 能把历史内容也解密出来，但密钥分发过程不能泄密 | 新成员的设备                                     |

基于这个拆解，会自然引出下面的三层密钥结构。

第一层：主题密钥（对称密钥）
负责加密所有记录内容。

每个加密主题创建时，自动生成一个随机对称密钥（比如 AES-256 密钥）。

这个主题下的所有文字、图片、音频，全部用这个密钥加密后再写入本地文件。

文件同步到云盘时，本身就是密文，所以传输过程不需要额外再做一套端到端加密——HTTPS 负责传输安全，密文文件负责存储安全。云盘只能看见一堆乱码文件，无法解密。

这就是你说的「传输过程中的加密」。本质上，文件在上传前已经用主题密钥加密好了，所以无论经过什么同步通道，看到的都是密文。

第二层：用户身份密钥（非对称密钥）
负责安全地把主题密钥分发给各个成员。

每个用户第一次使用 App 时，会在本地生成一对公私钥（例如 Curve25519），公钥就是你的 ID，私钥永远留在你的设备上。

主题创建者要把主题密钥分享给成员 A 时：

用 A 的公钥 加密「主题密钥」。
将加密后的密钥包（一个很小的文件）存到主题的元数据里，同样可以同步。
成员 A 的设备同步到这个密钥包后，用自己的私钥解密，就拿到了主题密钥。

之后 A 就能解密该主题所有的历史消息和文件。

这样一来，除了主题成员，任何第三方（包括云盘）都无法解开主题密钥，也就无法看到内容。

第三层：本地 passphrase（你提出的那个问题）
负责保护用户私钥和已解密的主题密钥在本地存储的安全。

当新用户加入一个加密主题后，他本地已经拿到了用他公钥加密过的主题密钥，然后用自己的私钥解密，得到了明文主题密钥。
但是，这个明文主题密钥以及你的私钥本身，在手机里不能明文躺着。否则任何人拿到你的手机就可以直接读取所有记录。

所以需要再加一把本地的「锁」——这个锁就是你设置的 passphrase（或者指纹/面容解锁，最终也是由 passphrase 保护）。

推荐的做法是：

每个用户设置自己的 passphrase，不同用户完全独立。

用户的私钥和解密后的主题密钥，在存储到本地数据库或 keychain 前，用一个由 passphrase 派生出来的密钥（通过 KDF 如 Argon2）加密保存。

每次打开 App 需要解密时，输入 passphrase 才能解锁密钥。

为什么不用全局 passphrase？
如果所有成员共用一个全局 passphrase，就等于这把锁的钥匙多人持有，泄露风险成倍增加，而且任何一个人改密码都会影响所有人。各自设置自己的 passphrase 更安全，也符合「数据属于自己」的原则。

完整流程串联起来
建主题
→ 生成主题密钥，用创建者自己的公钥加密后存入元数据，所有文件用主题密钥加密落盘。

同步 / 传输
→ 加密文件通过云盘同步，HTTPS 保证传输通道安全，文件本身已经是无法解密的密文。

加人
→ 邀请人用新成员的公钥加密一份主题密钥，上传。新成员同步到后用自己私钥解密出主题密钥。

新成员本地保护
→ 要求新成员设置自己的 passphrase。用这个 passphrase 加密存储他的私钥和主题密钥，避免明文落地。

自行解密、不依赖 App
→ 用户随时可以导出：

自己的私钥（由 passphrase 保护的备份文件）

主题密钥（明文或用自己的 passphrase 加密）
→ 然后用任意支持 AES-256 的工具和密钥，自行解密那些同步出去的密文文件。

### 如何做到真正的“启动快”？

既然是 IM 样式，随着记录增加，数据库和本地文件会急速膨胀。

优化方案：

UI 骨架屏与懒加载： 启动时，Tab 1 的 Channel List 必须进行内存缓存（MMKV 或 Room Pre-fetch），前 20 条消息做到毫秒级渲染。

发送框常驻/小部件： 针对“不错过瞬间思绪”的诉求，建议提供 Android 桌面微件（Widget）或通知栏常驻快捷入口，点击直接拉起发送框，连进入 App 寻找 Channel 的步骤都省去（可默认发送到“闪念/未分类”频道）。

### 文件系统选择

Android 的 Scoped Storage（分区存储）对公开文件系统限制严格。如果文件存放在 Storage/Shared/YourApp，需要处理 MediaProvider 的刷新问题，否则系统相册可能扫不出你下载的图片。

### Android 技术栈（可选）：

数据库（Meta）： SQLDelight 或 Room。如果配合 PowerSync，它原生支持 SQLite。

存储加密： SQLCipher（针对数据库加密）；媒体文件使用 AES-GCM 算法流式加密，密钥可由用户自定义的 Passphrase 通过 PBKDF2 生成，不依赖任何云端。

异步与流： 消息列表的实时刷新完全基于 Kotlin Flow，做到数据源（Database）变动，UI 自动 Recomposition。

### 消息列表界面的「音频交互」
  
核心交互：气泡内“波形化”播放器
传统的“小喇叭+秒数”图标太古老，且无法预判语音内容。最佳实践是渐变色波形图（Audio Waveform）。

```text
+---------------------------------------------------+
| [▶]  |||i||i||i|i|||i||i||i|i|||i||i||i|   0:42  | -> 未播放状态（波形暗色）
+---------------------------------------------------+

+---------------------------------------------------+
| [⏸]  ███████████|i||i||i|i|||i||i||i|   0:15  | -> 播放中（已播放部分高亮）
+---------------------------------------------------+
```
  
交互细节：

静态波形提取： 在录音生成文件时，同时计算并存储一组低维度的振幅数据（如一个 List<Float> 数组，30-50个点）。消息渲染时，Compose 根据这组数据绘制 Canvas 波形。

点击/拖拽进度（Seekable）： 用户点击波形图的任意位置，可以直接进度跳转（Seek）；按住波形左右滑动，可以丝滑地快进/快退。

视觉反馈： 随着播放进度推进，波形颜色从左到右“被染色”高亮。

二、 高级进阶交互（针对记录软件的痛点）

1. 倍速播放（Speed Control）—— 提效神器
交互： 当点击播放语音后，气泡旁或播放按钮旁静默出现一个 1X 的小标签。

  操作： 点击在 1X -> 1.5X -> 2X 之间循环切换。

  目的： 用户在复盘自己或他人的长语音灵感时，2倍速能节省大量时间。

2. 全局悬浮播放条（Global Audio Player）—— 跨界面浏览
痛点： 用户点开一条 3 分钟的语音，听的过程想退出当前主题，去“查看（Flow）” Tab 翻阅去年的记录，或者去另一个主题查看资料。如果一退出聊天框语音就断了，体验会非常糟糕。

  交互：

  当语音播放时，用户退出当前主题，界面顶部（TopAppBar 下方）或底部优雅地弹出一个极简的悬浮播放条。

  包含：当前播放的主题 Icon、进度条、播放/暂停按钮、关闭按钮。

  点击悬浮条，可以瞬间反向跳转回该语音所在的主题和具体消息位置。

3. “听筒/扬声器”智能切换与标记
近接传感器（Proximity Sensor）监听： 手机贴近耳朵时，自动从扬声器切换为听筒播放，并适当降低音量（保护隐私/防打扰）。

  已听/未听状态： 协同状态下，别人发的语音在未听时气泡右上角有一个“小蓝点”，播放完后蓝点消失。

### 数据库同步的概念

在本地优先（Local-First）的架构中，数据存在三个层次的生命周期，而不仅仅是“同步”与“不同步”：
共享层（Shared）：多人之间同步（如：谁是成员、发了什么消息）。
个人同步层（Personal Synced）：我自己的多台设备间同步，别人看不到（如：我是否置顶、我是否归档）。
设备专属层（Device Local）：完全不参与同步，只留在当前这台手机或平板上（如：针对当前屏幕大小适配的字体、未发送的草稿、本地缓存状态）。
